# Metodologia de Análise de Requisitos para um Sistema de Gerenciamento de Biblioteca

### Introdução
A escolha da metodologia de análise de requisitos é um pilar fundamental para o sucesso de qualquer projeto de desenvolvimento de software. Ela define o arcabouço pelo qual as necessidades dos usuários e as funcionalidades do sistema serão identificadas, documentadas, validadas e gerenciadas. Para um Sistema de Gerenciamento de Biblioteca (SGB), que envolve múltiplos stakeholders e um conjunto diversificado de funcionalidades, a adoção de uma abordagem estruturada é essencial para garantir que o sistema final seja robusto, eficiente e alinhado às expectativas. Esta seção descreve a metodologia proposta para a análise de requisitos do SGB, combinando elementos de abordagens tradicionais e ágeis para otimizar o processo.

# Metodologia Proposta: Abordagem Híbrida e Iterativa
Para a análise de requisitos do Sistema de Gerenciamento de Biblioteca, propõe-se uma abordagem híbrida e iterativa, que integra a estrutura e a documentação detalhada de metodologias tradicionais com a flexibilidade e a adaptabilidade das metodologias ágeis. Esta escolha visa capitalizar os pontos fortes de ambas as abordagens, mitigando seus respectivos desafios.

### Fases da Metodologia
A metodologia será dividida nas seguintes fases principais, com ciclos iterativos dentro de cada uma:

##### 1.Elicitação de Requisitos:
Objetivo: Coletar as necessidades e expectativas de todos os stakeholders.
Técnicas: Serão utilizadas diversas técnicas de elicitação, como entrevistas com bibliotecários, usuários e gerência; análise de documentos existentes (manuais, relatórios, políticas); observação direta dos processos da biblioteca; workshops e sessões de brainstorming com grupos de stakeholders; e questionários para coletar feedback de um público mais amplo. O benchmarking com outros sistemas de biblioteca também será empregado para identificar melhores práticas e funcionalidades padrão.

##### 2.Análise e Modelagem de Requisitos:
Objetivo: Refinar, organizar e representar os requisitos de forma clara e não ambígua.
Técnicas: Os requisitos brutos coletados serão analisados para identificar inconsistências, ambiguidades e lacunas. Serão utilizados modelos como:
Casos de Uso: Para descrever as interações entre os atores e o sistema, detalhando as funcionalidades sob a perspectiva do usuário.
Diagramas de Classes (UML): Para modelar a estrutura estática do sistema, mostrando as entidades (livros, usuários, empréstimos) e seus relacionamentos.
Diagramas de Atividades/Fluxo de Dados (DFD): Para representar o fluxo de informações e os processos de negócio dentro do sistema.
Protótipos (de baixa e média fidelidade): Para visualizar a interface do usuário e as interações, permitindo que os stakeholders forneçam feedback precoce sobre o design e a usabilidade.

##### 3.Especificação de Requisitos:
Objetivo: Documentar formalmente os requisitos funcionais e não funcionais.
Documento de Especificação de Requisitos (DER): Será elaborado um documento detalhado contendo todos os requisitos, suas prioridades, critérios de aceitação e rastreabilidade. Os requisitos funcionais serão descritos de forma clara e verificável, enquanto os requisitos não funcionais abordarão aspectos como desempenho, segurança, usabilidade e escalabilidade.

#####  4.Validação de Requisitos:
Objetivo: Garantir que os requisitos documentados estejam corretos, completos, consistentes e atendam às necessidades reais dos stakeholders.
Técnicas: Revisões formais e informais com os stakeholders, prototipagem interativa e testes de aceitação de usuário (UAT) em fases iniciais com base nos protótipos serão realizados para validar os requisitos. Isso ajuda a identificar e corrigir erros antes que se tornem mais caros de resolver.

##### 5.Gerenciamento de Requisitos:
Objetivo: Controlar as mudanças nos requisitos ao longo do ciclo de vida do projeto.
Processo de Mudança: Será estabelecido um processo formal para solicitar, analisar, aprovar e implementar mudanças nos requisitos. Ferramentas de gerenciamento de requisitos podem ser utilizadas para manter a rastreabilidade e o controle de versão dos requisitos.

# Justificativa da Abordagem Híbrida
A escolha de uma abordagem híbrida é justificada pela natureza do projeto de um SGB. Embora existam funcionalidades centrais bem definidas (gestão de acervo, empréstimos), há também a necessidade de flexibilidade para incorporar feedback dos usuários e adaptar-se a novas demandas ao longo do tempo. A estrutura das metodologias tradicionais garante uma base sólida e documentada, enquanto a iteração e o feedback contínuo das metodologias ágeis permitem a adaptação e a entrega de valor incremental.

# Conclusão
A metodologia de análise de requisitos proposta, com sua abordagem híbrida e iterativa, visa estabelecer um processo robusto e flexível para o desenvolvimento do Sistema de Gerenciamento de Biblioteca. Ao enfatizar a elicitação abrangente, a modelagem detalhada, a especificação clara, a validação contínua e o gerenciamento eficaz de mudanças, garantimos que o SGB será construído sobre uma base sólida de compreensão das necessidades. Isso resultará em um sistema que não apenas atende aos requisitos técnicos, mas que também proporciona uma experiência de usuário superior e um valor duradouro para a instituição.
